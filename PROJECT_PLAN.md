# План проекта: медицинский центр (React + Django + PostgreSQL)

Документ фиксирует целевую структуру репозитория и этапы развития. Код на этом этапе не создаётся — только архитектурная схема.

---

## 1. Корень репозитория

Родительская папка проекта **разделяется на две независимые части**:

| Папка   | Назначение                          |
|---------|-------------------------------------|
| `back/` | Backend: Django, API, работа с БД   |
| `front/`| Frontend: React, UI, запросы к API  |

В корне допустимы общие для всего монорепозитория файлы: `README.md`, этот план, при необходимости — `docker-compose.yml`, `.gitignore`, документация по деплою, скрипты оркестрации (например, отдельный скрипт деплоя бекенда, если он живёт на уровне корня).

**Принцип:** `back` и `front` — отдельные проекты со своими зависимостями и своими точками входа; связываются по HTTP (REST или иной договорённый API).

---

## 2. Структура `back/` (Django + PostgreSQL)

Рекомендуемый каркас (имена приложений Django можно уточнить под предметную область):

```
back/
├── manage.py
├── requirements.txt          # или pyproject.toml + poetry/pip-tools
├── .env.example              # шаблон переменных (без секретов в git)
├── config/                   # или <project_name>/
│   ├── settings/
│   │   ├── base.py
│   │   ├── development.py    # опционально, если нужна локальная разработка
│   │   └── production.py
│   ├── urls.py
│   ├── wsgi.py
│   └── asgi.py
├── apps/                     # доменные приложения Django
│   ├── accounts/             # пользователи, роли, аутентификация (при необходимости)
│   ├── core/                 # общие модели, утилиты, базовые миксины
│   └── <domain>/             # сущности медицинского центра (записи, врачи, услуги и т.д.)
├── static/                   # при необходимости для админки/сбора статики
├── media/                    # загрузки пользователей (в git не коммитить)
└── tests/                    # интеграционные тесты API (опционально, если не внутри apps)
```

**База данных**

- СУБД: **PostgreSQL** (отдельный сервис на сервере, как у вас принято).
- Подключение через переменные окружения (`DATABASE_URL` или набор `DB_HOST`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_PORT`).
- Миграции Django — единственный источник правды по схеме в коде.

**API**

- Типичный стек: **Django REST Framework** для JSON API.
- Версионирование URL (`/api/v1/...`) — по мере роста проекта.
- Аутентификация: JWT или session + CSRF для same-site; для SPA чаще JWT + refresh или cookie-based session с настройкой CORS.

**Безопасность и эксплуатация**

- Секреты только в окружении сервера / `.env`, не в репозитории.
- `ALLOWED_HOSTS`, `CORS_ALLOWED_ORIGINS` под домены фронта.
- Деплой бекенда — по вашему процессу (отдельный скрипт в корне репозитория, например `deploy_backend.py`, если он появится).

## 3. Структура `front/` (React)

Рекомендуемый каркас (сборщик: **Vite + React + TypeScript** — современный стандарт; при желании можно зафиксировать альтернативу в README):

```
front/
├── package.json
├── package-lock.json / pnpm-lock.yaml
├── vite.config.ts
├── tsconfig.json
├── index.html
├── .env.example              # VITE_API_BASE_URL и прочие публичные префиксы
├── public/
└── src/
    ├── main.tsx
    ├── App.tsx
    ├── assets/
    ├── components/           # переиспользуемые UI-компоненты
    ├── pages/                # экраны/маршруты
    ├── layouts/
    ├── hooks/
    ├── services/             # обёртки над fetch/axios к API
    ├── store/                # при использовании Redux/Zustand и т.п.
    ├── types/                # TypeScript-типы, в т.ч. договорённые с API
    └── routes/               # конфигурация маршрутизации (React Router)
```

**Связь с бекендом**

- Базовый URL API задаётся переменной окружения (например, `VITE_API_BASE_URL`), чтобы один код собирался под dev/stage/prod.
- Единый слой `services/` (или `api/`) избегает размазывания URL по компонентам.

## 4. Взаимодействие front ↔ back

1. Браузер загружает SPA из `front` (статический хостинг или тот же nginx, что отдаёт `index.html`).
2. Запросы идут на домен/путь API бекенда (`back`), например `https://api.example.com/api/v1/...`.
3. Django обрабатывает запрос, обращается к PostgreSQL, возвращает JSON.
4. CORS на бекенде разрешает origin фронта.

Документирование контракта: OpenAPI/Swagger (через `drf-spectacular` или аналог) — желательно на этапе стабилизации API.

## 5. Этапы внедрения (roadmap)

| Этап | Содержание |
|------|------------|
| 0 | Инициализация: `back` — `django-admin startproject`, настройка PostgreSQL через env; `front` — шаблон Vite React TS. |
| 1 | Базовый health-check endpoint, CORS, пример одной сущности (модель + миграция + DRF ViewSet). |
| 2 | Аутентификация и авторизация (роли согласно требованиям медицинского центра). |
| 3 | Основные доменные модули в `apps/`, постепенное наполнение API. |
| 4 | Экраны на React, формы, валидация, обработка ошибок API. |
| 5 | Тесты (pytest на бекенде, Vitest/RTL на фронте), CI при необходимости. |
| 6 | Деплой и мониторинг (логи, резервное копирование БД — вне этого файла, по вашей инфраструктуре). |
